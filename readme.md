## C++ App Overview

### What the app does

The service exposes four endpoints:
- GET /health
- GET /version
- GET /config
- POST /echo

For Milestone 1, thatÔÇÖs enough to demonstrate:
- buildable C++ service
- local config via environment variables
- unit-testable logic
- integration testing over HTTP

---

### What CMake is doing here

`CMakeLists.txt` is the build recipe.

At a high level it does four things:

#### 1. Defines the project

It sets:
- project name
- version
- C++ standard (`C++17`)

---

#### 2. Pulls external dependencies

It uses `FetchContent` to download:
- `cpp-httplib` Ôćĺ lightweight HTTP server library
- `nlohmann/json` Ôćĺ JSON library
- `googletest` Ôćĺ unit test framework

That means:
- you do not install those libraries manually with `apt`
- CMake fetches them during configure/build

---

#### 3. Builds targets

It creates:
- `echo_core` Ôćĺ shared internal app logic
- `echo_server` Ôćĺ the actual executable binary
- `unit_tests` Ôćĺ the C++ test binary

---

#### 4. Registers tests

It enables:
- C++ unit tests through `ctest`

Important distinction:
- C++ unit tests are managed by **CMake/CTest**
- Python integration tests are run separately by **pytest**

---

### What the Python integration test does

`app/tests/integration/test_api.py` is an external black-box test of the running service.

It does this:

#### 1. Looks for compiled binary

It expects:
```
build/echo_server
```
So first you must build the app.

---

#### 2. Starts the server as a subprocess

The test fixture launches the binary with test environment variables such as:
- bind address
- port
- environment
- node id

---

#### 3. Waits until the server is reachable

It polls `/health` until the service responds.

---

#### 4. Executes real HTTP calls

It tests:
- `/health`
- `/version`
- `/echo`
- invalid `/echo` payload

---

#### 5. Stops the process

After tests finish, it terminates the server.

That means the integration test is very realistic:
- compile binary
- start real service
- test over HTTP
- stop service

---

### What each script does

#### `scripts/bootstrap.sh`

*Purpose*: prepare Python side.

It does:
- create virtual environment .venv
- activate it
- upgrade pip
- install Python dependencies from tools/python_helper/requirements.txt

For Milestone 1, those dependencies are mainly:
- pytest
- requests

So despite the folder name `python_helper`, at this stage it is basically the Python environment for integration tests.

---

#### `scripts/build.sh`

*Purpose*: configure and compile project.

It does:
- `cmake -S . -B build ...`
- `cmake --build build --parallel`

Meaning:
- source dir = current repo
- build dir = `build/`
- generate build system files
- compile targets

---

#### `scripts/test.sh`

*Purpose*: run all tests for Milestone 1.

It does:
1. `ctest --test-dir build --output-on-failure`
 - runs C++ unit tests registered by CMake
2. `pytest app/tests/integration -q`
 - runs Python integration tests

So this script is the single entry point for ÔÇťverify app locallyÔÇŁ.

---

#### `scripts/run_local.sh`

*Purpose*: start the app locally without systemd.

- starts the compiled binary from `build/echo_server`
```
build/echo_server
```
- uses environment variables for configuration
  - `APP_BIND_ADDRESS`
  - `APP_PORT`
  - `APP_ENV`
  - `APP_NODE_ID`
- intended for quick local development and debugging

By default it runs on:
- 127.0.0.1
- port 8080

---

#### `scripts/install_service.sh`

*Purpose*: install or update the application as a systemd service.

- copies the compiled binary to `/opt/echo-app/`
- creates or updates environment configuration at `/etc/echo-app/app.env`
- creates or updates systemd unit file `/etc/systemd/system/app.service`
- reloads systemd and enables the service
- restarts the service to apply updates

The application is configured via:
```
/etc/echo-app/app.env
```

Example:
```
APP_BIND_ADDRESS=0.0.0.0
APP_PORT=8080
APP_ENV=dev
APP_NODE_ID=node-local
```

You can modify this file and restart the service:
```
sudo systemctl restart app.service
```

---

### `scripts/service_control.sh`

*Purpose*: control the systemd service lifecycle.

Supported commands:

- `start` Ôćĺ start the service
- `stop` Ôćĺ stop the service
- `restart` Ôćĺ restart the service
- `status` Ôćĺ check service status
- `logs` Ôćĺ stream service logs via `journalctl`
- `enable` Ôćĺ enable service at boot
- `disable` Ôćĺ disable service at boot

Example:

```bash
./scripts/service_control.sh status
./scripts/service_control.sh logs
```

### Linux packages you should install

Because dependencies are fetched by CMake, system packages are fairly minimal.

On Ubuntu/Debian, install:
```
sudo apt update
sudo apt install -y \
  build-essential \
  cmake \
  git \
  curl \
  python3 \
  python3-venv \
  python3-pip
```

That should be enough for Milestone 1.

Optional but useful later:
```
sudo apt install -y jq
```

You do not strictly need jq right now.
